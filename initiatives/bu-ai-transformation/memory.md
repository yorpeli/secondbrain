## The Squad Model — Where We Landed
- **Shape**: 1 PM-Builder + 1–2 AI Engineers per squad.
- **Home**: BU AI Transformation pillar. Squads do NOT transfer into BUs — they rotate.
- **Funding**: Two stripes — ~20–30% base (pillar-funded, protects platform contribution and learning loop) + ~70–80% engagement (BU-funded for BU work). Mechanism (chargeback / slot allocation) is the Finance dependency.
- **Ownership**: BU owns workflow, metrics, adoption, outcome. Squad owns AI-native design, build, evaluators, governance hooks, component harvest.
- **Initial ask**: 2 squads (4 people) to start. Scale gated on intake board demand, not calendar.

## Why the Squads (Not Consulting) Argument
1. Value unit, not headcount (from AIR² Leap deck §01).
2. Skip Augmented, go AI-Native — BUs alone will land in copilot territory.
3. Forcing function — every build leaves a sharper evaluator behind. Compounding is enforced.
4. Talent scarcity — pool the rarest people, rotate, retain.
5. Compounding economics — the second BU is cheaper than the first only if the same builders ship both. Independent builds = no reuse, fragmented audit surfaces, no demand signal for AI Foundations.

## Reframing John & Oren
- John's "function owns AI outcomes / funds and resources execution" → function PAYS for the build; it does not have to STAFF the build. Resourcing happens via the squad model.
- Oren's "Platform provides shared foundations / reusable capabilities" → those foundations get authored bottom-up from real builds. Squads are the mechanism. Without them, AI Foundations is a roadmap looking for customers.
- Phase 0 memo's "support and accelerate through enablement, tooling, infrastructure, and guidance" → guidance does not close the execution gap. Build capacity does. Three swaps: support → build capacity; BU-staffs → BU-funds; customers-of-advice → owners-of-outcomes-contracting-for-capacity.
